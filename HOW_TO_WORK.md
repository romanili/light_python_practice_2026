# Как работать с репозиторием

Связанные документы:
- [`README.md`](README.md) — общий обзор практики и выбор варианта.
- [`project_full.md`](project_full.md) — полный вариант.
- [`project_full_easy.md`](project_full_easy.md) — упрощенный вариант.
- [`project_light.md`](project_light.md) — базовый вариант.
- [`SQLITE_GUIDE.md`](SQLITE_GUIDE.md) — рекомендации по `sqlite3` и инициализации базы.

## Базовый порядок
1. Сделать fork репозитория на GitHub.
2. Склонировать форк локально.
3. Использовать папку `src/` для кода.
4. Заполнить `student_meta.md` внутри `src/`.
5. Работать по этапам из выбранного варианта.
6. Писать журнал работ.
7. Делать коммиты и пушить изменения в GitHub.

## `src/`
- Весь код проекта хранить внутри `src/`.
- Точка входа проекта — `src/main.py`.
- Остальную логику можно выносить в отдельные модули внутри `src/`.
- В `src/` должен лежать `student_meta.md`, который студент заполняет в форке.

## Метаинформация
Студент в форке заполняет:
- группу;
- ФИО;
- вариант: `полный`, `упрощенный` или `базовый`.

## Ограничения по коду
- В основе должна быть стандартная библиотека Python.
- Высокоуровневые библиотеки и фреймворки не использовать.
- В полном варианте SQLite обязателен, использовать только `sqlite3`.
- В упрощенном и базовом вариантах SQLite не использовать.
- Не заменять собственную реализацию готовой оберткой, которая скрывает основную работу.
- Для полного варианта схему SQLite можно держать в коде, база создается при запуске утилиты.

## Журнал
Журнал нужен, чтобы кратко фиксировать:
- что сделано;
- что не получилось;
- что осталось на следующий этап.

### Формат записи
- дата;
- этап;
- короткий результат;
- проблемы;
- следующий шаг.

### Правило
Запись должна быть короткой и полезной, без лишнего текста.

## Коммиты
- Один коммит — одна логическая доработка.
- Сообщение должно быть коротким и понятным.
- Не смешивать в одном коммите разные этапы.

## Пуши
- После завершения заметной части работы пушить изменения в GitHub.
- Не копить слишком большой незапушенный хвост.

## Что сдавать
- код в `src/`;
- заполненную метаинформацию;
- выполненный этап по выбранному плану;
- короткую запись в журнале;
- актуальные коммиты в GitHub.
